Problema e escolha técnica

Broadcast massivo exige canal correto e controle de taxa

Abordagem Realidade em alto volume Uso recomendado
Power Automate Limites por licença/conector; não projetado para processamento massivo; pode sofrer throttling e latência. Fluxos de automação e integrações de menor escala.
Microsoft Graph Throttling multi-nível: app, tenant e user. Bom para integração, não para broadcast massivo. Operações direcionadas e integrações com Microsoft 365.
Bot Framework Canal nativo para notificações e proactive messaging; melhor suporte para escala, com rate limits dinâmicos. Mensagens proativas 1:1 após instalação do Teams App.

// API fan-out
app.post("/api/send", requireApiKey, async (req, res) => {
  const validation = validateMessage(req.body?.message);
  const jobId = uuidv4();
  await createJob(jobId, validation.serialized, total);

  for await (const ref of streamRefs()) {
    batch.tryAddMessage({
      body: { jobId, refJson: ref.refJson, rowKey: ref.rowKey },
      messageId: `${jobId}:${md5(rowKey)}:${repeatIndex}`
    });
  }
});
// Worker delivery
if (RATE_LIMIT_ENABLED) {
  await acquireToken(); // Redis Lua token bucket
}

await adapter.continueConversation(ref, async (ctx) => {
  await deliver(ctx, msg); // text or Adaptive Card
});

Resumo para decisão técnica

Solução proposta: esteira de mensagens proativas, controlada e observável

Canal adequado

  • Bot Framework proactive messaging.
  • Entrega 1:1 após instalação do Teams App.
  • Suporte a texto e Adaptive Cards.

Arquitetura elástica

  • ACA API + ACA Worker.
  • Service Bus para fan-out e DLQ.
  • KEDA scale-to-zero.

Controle operacional

  • Redis counters e token bucket.
  • Health/readiness endpoints.
  • Status de job via API.

Resultado: broadcast em massa sem chamadas síncronas por usuário, com capacidade planejável e controle explícito de throttling.
